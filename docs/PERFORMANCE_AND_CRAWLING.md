# Phase 1 — Performance & Crawler Access (plain-language guide)

This is the foundation phase: **AI engines and search engines cannot rank or
cite content they cannot load or read.** Here is what changed, why it matters,
and the authoritative source behind each decision.

## Jargon, defined once

- **Core Web Vitals (CWV)** — Google's three real-user speed/stability metrics.
  - **LCP** (Largest Contentful Paint): how long until the biggest thing on
    screen (usually the hero image) appears. **Good ≤ 2.5 s.**
  - **INP** (Interaction to Next Paint): how quickly the page reacts when you
    click/tap/type. **Good ≤ 200 ms.** (Replaced FID in March 2024.)
  - **CLS** (Cumulative Layout Shift): how much the page visually jumps around
    while loading. **Good ≤ 0.1.**
  - A page "passes" only if **75% of real visits** are good on all three.
  - Source: [web.dev/articles/vitals](https://web.dev/articles/vitals),
    [Google Search Central: Core Web Vitals](https://developers.google.com/search/docs/appearance/core-web-vitals).
    (Note: a rumored 2026 tightening of LCP to 2.0 s is **not** in Google's
    official docs; 2.5 s remains current.)
- **Lab vs. field data** — "Lab" = a single test in a controlled environment
  (what our CI runs). "Field" = anonymized data from real Chrome users (CrUX).
  **Field is the real scoreboard;** lab is only for catching regressions early.
  Source: [web.dev: lab and field data](https://web.dev/articles/lab-and-field-data-differences).

## What changed in this phase

### 1. Explicit AI-crawler policy (`public/robots.txt`)

- **What:** `robots.txt` now explicitly **welcomes** AI crawlers — GPTBot &
  OAI-SearchBot (OpenAI/ChatGPT), ClaudeBot (Anthropic), PerplexityBot,
  Google-Extended (Google AI), Applebot-Extended (Apple).
- **Why:** these bots are how your site can appear in and be **cited by** AI
  answers. They were already allowed implicitly (`User-agent: *`); making it
  explicit documents the intent and gives a one-line opt-out per bot.
- **Trade-off (your call, already chosen):** allowing them means your public
  content can be used by these AI systems. You chose to allow — the recommended
  default for discoverability.

### 2. Performance budget in CI (`lighthouserc.json` + `.github/workflows/ci.yml`)

- **What:** every pull request now runs **Lighthouse CI** three times per page
  and takes the **median** (single runs on shared CI machines are noisy). It
  **hard-fails** the build on the lab-stable Core Web Vitals:
  - LCP > 2.5 s → **fails** the build
  - CLS > 0.1 → **fails** the build
  - Total Blocking Time > 300 ms → **warns** (advisory). TBT is Lighthouse's
    **lab proxy for INP**, but INP can only be measured from **real users**
    ([web.dev/articles/inp](https://web.dev/articles/inp)), and lab TBT on a
    shared CI runner is noisy — so we report it here and watch true **INP in the
    field** (CrUX) after deploy, rather than hard-gate on a noisy proxy.
  - Performance / Accessibility / SEO scores below 0.9 → **warn** (reported).
- **Why this split:** it enforces the metrics that are reliable in a lab (layout
  stability and load speed) while honestly treating the one that isn't
  (interaction responsiveness) as field-measured. This matches the standards'
  own guidance that **lab is diagnostic and field is the scoreboard.**
- **Honest note:** the homepage ships only ~16 KB of JavaScript (one View
  Transitions module), so it is already JS-light — a first noisy single-run
  measurement reported an inflated 1183 ms TBT, which the 3-run median corrects.

## What was already good (no change needed — verified)

The upstream template was already well-built here, so we did **not** invent work:

- **Server-rendered content:** Astro `output: 'static'` means all page content
  is in the HTML — crawlers and LLMs read it without running JavaScript.
- **LCP image:** the hero already uses `loading="eager"` + `fetchpriority="high"`
  - responsive `widths`/`sizes` (the web.dev-recommended LCP pattern).
- **CLS protection:** the `<Image />` component sets explicit dimensions, which
  reserves space and prevents layout jumps.
- **Fonts:** Inter is **self-hosted** (via Fontsource) with `font-display: swap`
  — no third-party request, no invisible-text delay.
- **Sitemap:** generated by `@astrojs/sitemap`, and its link is auto-appended to
  `robots.txt` at build time by the template integration.

## How to read the results once live

After deploy, check real-user (field) data — the true measure:

- **PageSpeed Insights** (`pagespeed.web.dev`) — paste your URL; the top
  "Discover what your real users are experiencing" panel is CrUX field data.
- **Google Search Console → Core Web Vitals** report — once the site has enough
  traffic, this shows pass/fail trends over time.

Lab numbers in CI will look slightly different from field — that's expected and
documented above.
